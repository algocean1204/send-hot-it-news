Flutter 앱은 별도 API 서버 없이 SQLite 파일을 직접 읽기/쓰기합니다. Python 백엔드도 같은 DB 파일에 접근합니다. (WAL 모드로 동시 접근 안전)

1

홈 (Overview)

▶

봇의 전체 상태를 한눈에 확인하는 대시보드 메인 화면. 읽기 전용으로 DB를 직접 조회합니다.

기능 리스트

  • 봇 상태 표시 (마지막 실행 시각, 성공/실패)
  • 오늘 전송 건수 집계
  • 최근 에러 요약 (최근 N건)
  • 다음 실행 시간 계산 (launchd 스케줄)
  • 구독자 수 표시 (대기/승인 각각)

연결 모듈 (Python 블럭)

직접 호출하는 모듈 없음 (읽기 전용, DB 직접 조회)

다음 실행 시간은 launchd plist 스케줄 기반 코드 로직으로 계산

연결 테이블

run_history processed_items error_log subscribers

데이터 흐름 방향

run_history ← R
processed_items ← R
error_log ← R
subscribers ← R
2

날짜별 뉴스

▶

30일치 뉴스를 날짜별로 브라우징하고, 핫뉴스를 수동 토글하는 화면. 읽기 + 쓰기 모두 수행합니다.

기능 리스트

  • 30일치 뉴스 날짜별 브라우징
  • 일반 뉴스 / 핫뉴스 탭 분리
  • 핫뉴스 영구 보관 (hot_news 테이블)
  • 뉴스 상세: 제목, 요약, 태그, 원문 링크, 파이프라인 경로, 소요 시간
  • 핫뉴스 수동 토글 (is_hot 업데이트 + hot_news INSERT/DELETE)

연결 모듈 (Python 블럭)

HotNewsDetector (#11)

핫뉴스 판단 기준 참조 (업보트 임계값, 소스 자동 태그 조건)

연결 테이블

processed_items hot_news

데이터 흐름 방향

processed_items ↔ RW
hot_news ↔ RW

핫뉴스 토글 시: processed_items.is_hot UPDATE + hot_news INSERT/DELETE

3

구독자 관리

▶

텔레그램 봇 구독자를 승인/거부/삭제 관리하는 화면. 구독자 데이터를 읽고 상태를 변경합니다.

기능 리스트

  • 대기(pending) / 승인(approved) / 거부(rejected) 탭
  • 검색 기능 (username, chat_id)
  • 승인 / 거부 / 삭제 버튼
  • 구독자 상세 (신청일, 승인일, 수신 이력)

연결 모듈 (Python 블럭)

SubscriberPoller (#3)

SubscriberPoller가 getUpdates로 수집한 데이터의 소스 테이블

연결 테이블

subscribers

데이터 흐름 방향

subscribers ↔ RW

읽기: 구독자 목록 조회 / 쓰기: status UPDATE, DELETE

4

실행 이력

▶

매시 정각 실행 결과를 시간순으로 확인하는 화면. 읽기 전용입니다.

기능 리스트

  • 매시 실행 결과 리스트
  • 수집/필터/요약/전송 건수
  • 총 소요시간, 모델 로드/추론 시간
  • 에러 발생 시 에러 메시지
  • 실행 상태 표시 (성공/부분실패/실패)

연결 모듈 (Python 블럭)

RunLogger (#14)

RunLogger가 기록한 실행 이력 데이터의 소스 테이블

연결 테이블

run_history

데이터 흐름 방향

run_history ← R
5

오류 로그 + 헬스체크

▶

오류 로그 조회 + 수동 헬스체크 실행 화면. 헬스체크 시 Python subprocess를 통해 HealthChecker 모듈을 직접 실행합니다.

기능 리스트

  • [오류 로그 탭] 시간순 에러 목록
  • [오류 로그 탭] 심각도별 필터 (severity)
  • [헬스체크 탭] 수동 실행 버튼
  • [헬스체크 탭] Ollama/소스/Telegram/DB/디스크 상태 리포트

연결 모듈 (Python 블럭)

HealthChecker (#17) ErrorNotifier (#15)

HealthChecker: Flutter에서 Python subprocess로 직접 실행 트리거
ErrorNotifier: error_log 테이블의 데이터 소스

연결 테이블

error_log health_check_results

데이터 흐름 방향

error_log ← R
health_check_results ← R

헬스체크 실행 시: Python subprocess가 health_check_results에 쓰기 → Flutter가 읽기

6

통계 대시보드

▶

소스별 수집 통계, 파이프라인 성공률, 응답 시간 추이 등을 차트로 시각화하는 화면. 통계 집계만 수행하므로 읽기 전용입니다.

기능 리스트

  • 소스별 건수 차트 (일별/주별)
  • 파이프라인 성공률 (APEX/Kanana/Claude)
  • 응답 시간 추이 (총 소요시간, 모델 로드/추론)
  • 필터링 효율 (수집 vs 통과 vs 전송 비율)

연결 모듈 (Python 블럭)

직접 호출하는 모듈 없음 (통계 집계용 DB 직접 조회)

연결 테이블

processed_items run_history

데이터 흐름 방향

processed_items ← R
run_history ← R

GROUP BY source, 날짜별 집계, pipeline_path 분석

7

설정 (Settings)

▶

소스 ON/OFF, 필터 임계값, 실행 제한 등 봇 설정을 관리하는 화면. 설정값을 읽고 변경합니다.

기능 리스트

  • 소스 ON/OFF 토글 (12개 소스 각각)
  • 필터 임계값 조정 (HN min_points, Reddit min_upvotes)
  • MAX_ITEMS_PER_RUN 조정
  • ALLOW_TIER1_OVERFLOW 토글
  • YOUNG_THRESHOLD_HOURS 조정

연결 모듈 (Python 블럭)

Filter (#7)

Filter 블럭이 다음 실행 시 변경된 설정값을 읽어 반영

연결 테이블

filter_config

데이터 흐름 방향

filter_config ↔ RW

읽기: 현재 설정 표시 / 쓰기: 사용자 변경 값 UPDATE

6. 누락 연결 점검

Flutter 화면에서 참조하지 않는 모듈

1. ConfigLoader — 정상: Python 내부 전용 (초기화)
2. MemoryGuard — 정상: Python 내부 전용 (메모리 체크)
4. Fetcher — 정상: Python 내부 전용 (데이터 수집)
5. Dedup — 정상: Python 내부 전용 (중복 제거)
6. LanguageDetector — 정상: Python 내부 전용 (언어 감지)
8. ModelEngine — 정상: Python 내부 전용 (LLM 엔진)
9. Summarizer — 정상: Python 내부 전용 (요약)
10. Translator — 정상: Python 내부 전용 (번역)
12. MessageFormatter — 정상: Python 내부 전용 (포맷팅)
13. TelegramSender — 정상: Python 내부 전용 (전송)
16. DataCleaner — 정상: Python 내부 전용 (데이터 정리)

위 11개 모듈은 Python 파이프라인 내부에서만 동작하므로 Flutter 화면 연결 없음이 정상입니다.

Flutter 화면에서 참조하는 모듈

3. SubscriberPoller — 화면 3 (구독자 관리) 데이터 소스
7. Filter — 화면 7 (설정) 설정 반영 대상
11. HotNewsDetector — 화면 2 (날짜별 뉴스) 판단 기준 참조
14. RunLogger — 화면 4 (실행 이력) 데이터 소스

연결 특이사항

화면 5 (헬스체크): HealthChecker를 Python subprocess로 직접 실행 — Flutter → Python 직접 트리거 (유일한 subprocess 호출)
화면 1 (홈): 다음 실행 시간은 DB가 아닌 launchd plist 스케줄 기반 코드 로직으로 계산
화면 2 (날짜별 뉴스): 핫뉴스 토글 시 processed_items + hot_news 2개 테이블 동시 변경 (트랜잭션 필요)

7. 데이터 보관 정책 요약

테이블 보관 기간 정리 블럭 정리 방식 관련 화면
processed_items 30일 DataCleaner (#16) 30일 초과 레코드 DELETE 화면 1, 2, 6
hot_news 영구 정리 대상 아님 - 화면 2
subscribers 영구 정리 대상 아님 수동 삭제만 (화면 3) 화면 1, 3
run_history 90일 DataCleaner (#16) 90일 초과 레코드 DELETE 화면 1, 4, 6
error_log 30일 DataCleaner (#16) 30일 초과 레코드 DELETE 화면 1, 5
filter_config 영구 정리 대상 아님 설정값, 삭제 불필요 화면 7
health_check_results 7일 DataCleaner (#16) 7일 초과 레코드 DELETE 화면 5

DataCleaner (#16)는 매시 정각 파이프라인 실행 시 마지막 단계에서 자동 실행되어 만료 데이터를 정리합니다. 영구 보관 테이블 (hot_news, subscribers, filter_config)은 DataCleaner의 정리 대상에서 제외됩니다.
