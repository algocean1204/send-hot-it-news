Hierarchical Module Design v1.0

news-pulse 계층적 모듈 설계서

12개 IT/AI 소스 → 로컬 LLM 요약/번역 → 텔레그램 푸시 봇 (시간당 1회)

생성일: 2026-04-08 총 블럭 수: 17개 (파이프라인 13 + 인프라 4) 기술스택: Python 3.12 · Ollama · LangChain · Flutter macOS · SQLite

목차

1. 기술스택 2. 프로젝트 폴더 구조 3. 전체 메인 파이프라인 4. 파이프라인 블럭 (13개) 5. 인프라 블럭 (4개) 6. ModelEngine 상세 구조 7. 폴백 체인 8. 공유 데이터 모델 9. 의존성 매트릭스 10. 구현 체크리스트

4. 파이프라인 블럭 — 13개

위치: src/ 각 하위 폴더 · 설계 원칙: 1블럭 = 1 Protocol, Strategy + DI, 공유 dataclass 통신

B1

ConfigLoader — 환경 설정 로더

▶
PROTOCOL
class ConfigLoader(Protocol): def load(self) -> Config: ...
IN (메인)
env_path: str ← 실행 인자 또는 기본값

.env 파일 경로 (기본: 프로젝트 루트)

OUT
반환: Config

봇 토큰, DB 경로, 소스 설정, 모델 설정 등 전체 설정 객체

Config { bot_token: str, # 텔레그램 봇 토큰 admin_chat_id: str, # 관리자 chat_id db_path: str, # SQLite 파일 경로 ollama_endpoint: str, # Ollama REST API 주소 apex_model_name: str, # APEX 모델 식별자 kanana_model_name: str, # Kanana 모델 식별자 memory_threshold_gb: float, # 메모리 임계값 (기본 26.0) sources: list[SourceConfig],# 12개 소스 설정 ... # 필터/보관 기간/핫뉴스 설정 }
내부 로직
  1. dotenv로 .env 파일 파싱
  2. TZ 이중 고정: os.environ['TZ'] = 'Asia/Seoul' + time.tzset()
  3. 필수 키 유효성 검증 (빈 문자열도 누락으로 처리)
  4. Config dataclass 생성 후 반환
의존 블럭: 없음 (파이프라인 첫 번째 블럭)
에러 처리: 필수 환경 변수 누락 → ValueError 발생, 파이프라인 즉시 중단
B2

MemoryGuard — 메모리 상태 확인

▶
PROTOCOL
class MemoryGuard(Protocol): def check(self, config: Config) -> MemoryStatus: ...
IN (메인)
config: Config ← B1.load()

메모리 임계값 설정 포함 (memory_threshold_gb: 26.0)

OUT
반환: MemoryStatus

Literal["local_llm", "claude_fallback"]

"local_llm" # 가용 RAM >= 26GB → 로컬 LLM 사용 "claude_fallback" # 가용 RAM < 26GB → Claude CLI 폴백
내부 로직
  1. psutil.virtual_memory().available로 가용 메모리 측정
  2. 임계값(26GB)과 비교하여 MemoryStatus 반환
의존 블럭: B1 ConfigLoader
에러 처리: psutil 호출 실패 → 안전하게 "claude_fallback" 반환 (로컬 LLM 시도 안 함)
B3

SubscriberPoller — 구독자 관리

▶
PROTOCOL
class SubscriberPoller(Protocol): def poll(self, config: Config) -> list[SubscriberEvent]: ...
IN (메인)
config: Config ← B1.load()

봇 토큰 포함

OUT
반환: List[SubscriberEvent]

이번 실행에서 감지된 구독 이벤트 목록. subscribers 테이블에 저장됨

SubscriberEvent { chat_id: str, # 텔레그램 chat_id username: str | None, # 사용자명 (없을 수 있음) event_type: str, # "subscribe" | "unsubscribe" occurred_at: datetime, # 이벤트 발생 시각 update_id: int # Telegram update_id (중복 방지) }
내부 로직
  1. Telegram Bot API getUpdates 호출 (offset으로 이미 처리된 건 건너뜀)
  2. /start 명령어 감지 → SubscriberEvent 생성
  3. subscribers 테이블에 pending 상태로 저장
의존 블럭: B1 ConfigLoader
에러 처리: API 호출 실패 → 빈 리스트 반환, 파이프라인 계속 진행
B4

Fetcher — 뉴스 수집기

▶
PROTOCOL
class Fetcher(Protocol): def fetch(self, config: Config) -> list[RawItem]: ...
IN (메인)
config: Config ← B1.load()

소스 목록, ON/OFF 설정, 타임아웃 설정

OUT
반환: List[RawItem]

수집된 원시 뉴스 아이템. 중복 포함, 필터링 전 상태

RawItem { url: str, # 원문 URL title: str, # 기사 제목 content: str | None, # 기사 본문 (없을 수 있음) source_id: str, # 소스 식별자 ("hackernews", "geeknews"...) fetched_at: datetime, # 수집 시각 upvotes: int | None, # 업보트 수 (Reddit/HN만) published_at: datetime | None, # 발행일시 url_hash: str # SHA256(url) }
구현체 (4개)
구현체담당 소스수집 방식
RssFetcherGeekNews, AI랩 블로그, Cursor ChangelogRSS/Atom 피드 파싱
AlgoliaFetcherHacker NewsAlgolia API (JSON)
RedditFetcherr/LocalLLaMA, r/ClaudeAI, r/Cursor.json 엔드포인트 (비인증)
GithubAtomFetcherClaude Code Releases, Cline ReleasesGitHub Atom 피드
내부 로직
  1. 활성화된 소스만 수집 대상으로 선별
  2. asyncio / ThreadPoolExecutor로 12개 소스 병렬 수집
  3. 각 소스별 Fetcher 구현체가 RawItem으로 정규화
  4. 모든 결과 합쳐서 List[RawItem] 반환
의존 블럭: B1 ConfigLoader
에러 처리: 개별 소스 실패 → 해당 소스 건너뜀, 나머지 수집 계속. 전체 타임아웃 초과 → 수집된 것만 반환
B5

Dedup — 중복 제거

▶
PROTOCOL
class Dedup(Protocol): def filter_new(self, items: list[RawItem]) -> list[NewsItem]: ...
IN (메인)
items: List[RawItem] ← B4.fetch()

Fetcher 출력 원시 아이템 (중복 포함)

OUT
반환: List[NewsItem]

중복 제거된 신규 아이템만. RawItem → NewsItem 변환 포함

NewsItem { url: str, title: str, content: str | None, source_id: str, fetched_at: datetime, upvotes: int | None, published_at: datetime | None, url_hash: str, lang: str # 아직 비어있음, LanguageDetector가 추가 }
구현체 (1개)
구현체설명
SqliteDedupprocessed_items 테이블에서 URL 해시(SHA256) 비교. 신규 항목은 즉시 삽입
의존 블럭: B1 ConfigLoader (DB 경로)
에러 처리: DB 접근 실패 → 전체 아이템을 신규로 간주 (중복 허용, 안전한 방향)
B6

▶
PROTOCOL

IN (메인)
item: NewsItem ← B5.filter_new()

lang 필드 미설정 뉴스 아이템

OUT
반환: NewsItem

lang 필드 추가됨 ("ko" | "en")

구현체 (2개, 체인)
구현체전략실행 조건

Filter — 계층화된 필터링

▶
PROTOCOL
class Filter(Protocol): def apply(self, items: list[NewsItem], config: Config) -> list[NewsItem]: ...
IN (메인)
items: List[NewsItem] ← B6.detect()

언어 감지 완료된 뉴스 아이템

IN (보조)
config: Config ← B1.load()

Tier별 임계값, 블랙리스트 키워드, MAX_ITEMS_PER_RUN(8) 설정

OUT
반환: List[NewsItem]

필터 통과 아이템. 최대 8건

구현체 (3개, 체인 순서대로)
순서구현체역할
1BlacklistFilter키워드 블랙리스트 전체 적용. 매칭 아이템 즉시 제거
2TierRouter소스별 Tier 분류 (1: AI랩+GitHub, 2: GeekNews, 3: Reddit+HN)
3PrioritySelectorTier별 우선순위 정렬 후 최종 8건 선택
Tier 정책
  1. Tier 1 (AI랩 블로그 + GitHub releases) — 무조건 통과
  2. Tier 2 (GeekNews) — 큐레이션 신뢰, 무조건 통과
  3. Tier 3 (Reddit + HN) — 업보트 임계값 + 연령 보정 (2시간 미만 = 완화)
  4. Tier 4 (전역) — 키워드 블랙리스트 (채용/광고/메타/밈)
  5. Tier 1+2가 8건 초과 시 그대로 허용 (ALLOW_TIER1_OVERFLOW)
의존 블럭: B1 ConfigLoader
에러 처리: 필터 체인 중 예외 → 해당 필터 건너뜀, 다음 필터 계속 적용
B8

ModelEngine — AI 모델 엔진

▶
PROTOCOL
class ModelEngine(Protocol): def load(self, model_name: str, keep_alive: int = 0) -> None: ... def generate(self, prompt: str, options: dict) -> str: ... def unload(self, model_name: str) -> None: ... def is_available(self) -> bool: ...
IN (메인)
model_name: str ← Config 또는 Summarizer/Translator

모델 식별자 (예: "APEX-4B", "kanana-1.5-8b")

prompt: str ← B9/B10 (Summarizer/Translator)

완성된 프롬프트 문자열

options: dict ← B9/B10

temperature, max_tokens 등 생성 옵션

OUT
반환: str

모델 응답 텍스트

구현체 (2개)
구현체방식특이사항
OllamaEngineOllama REST APIkeep_alive=0, load/generate/unload 순서 엄격 준수. 동시 적재 금지
ClaudeCliEngineclaude -p --bare subprocessload/unload 불필요, generate만 구현. is_available()은 which claude 확인
순차 로드 원칙: APEX 사용 완료 후 언로드 확인 → Kanana 로드. 동시에 두 모델 메모리 적재 금지
의존 블럭: B1 ConfigLoader
에러 처리: Ollama 연결 실패 → is_available() False 반환, Summarizer가 폴백 체인으로 처리. subprocess 실패 → CalledProcessError 발생
B9

Summarizer — 뉴스 요약기

▶
PROTOCOL
class Summarizer(Protocol): def summarize(self, item: NewsItem, engine: ModelEngine) -> SummaryResult: ...
IN (메인)

요약할 뉴스 아이템 (필터 통과)

IN (보조)
engine: ModelEngine ← 주입 (DI)

주입된 모델 엔진 (OllamaEngine 또는 ClaudeCliEngine)

OUT
반환: SummaryResult

요약 텍스트 + 메타 정보

SummaryResult { item_url: str, # 원문 URL summary_text: str, # 요약 텍스트 original_lang: str, # 원본 언어 ("ko" | "en") summarizer_used: str, # 사용된 Summarizer 구현체명 translator_used: str | None,# 번역기 (한국어 소스는 None) error: str | None # 에러 발생 시 메시지 }
구현체 (3개, FallbackChain)
순서구현체사용 모델특이사항
1ApexSummarizerAPEX-4B (OllamaEngine)한국어/영어 모두 요약. 한국어는 직접 한국어 요약
2KananaSummarizerkanana-1.5-8b (OllamaEngine)APEX 폴백. 단독 요약 시 사용

의존 블럭: B8 ModelEngine (주입)
에러 처리: 요약 실패 (타임아웃/파싱 오류) → FallbackChain이 다음 구현체 시도. 전체 폴백 실패 → error 필드 표시, 해당 아이템 전송 스킵

6. ModelEngine 상세 구조

ModelEngine (Protocol) ├── OllamaEngine │ ├─ load(model_name, keep_alive=0) → 모델 메모리 적재 │ ├─ generate(prompt, options) → str → Ollama REST API 추론 │ ├─ unload(model_name) → 즉시 메모리 해제 (keep_alive=0) │ └─ is_available() → bool → Ollama 서버 상태 확인 │ └── ClaudeCliEngine ├─ generate(prompt, options) → str → subprocess: claude -p --bare └─ is_available() → bool → which claude 확인 (load/unload 불필요 — 매 호출마다 subprocess 생성)
설계 핵심: Summarizer/Translator는 프롬프트 구성 + 응답 파싱만 담당한다. 실제 추론은 주입받은 ModelEngine에 위임한다. 엔진 교체 시 (OllamaEngine → vLLM 등) ModelEngine 구현체만 추가하면 된다.

8. 공유 데이터 모델 (dataclass)

블럭 간 통신은 이 dataclass 모델로만 이루어진다. 블럭은 서로를 모른다.

Config — 전체 설정 (봇 토큰, DB 경로, 소스, 모델, 필터 등) ▶
@dataclass class Config: # 텔레그램 bot_token: str admin_chat_id: str # DB db_path: str # 모델 ollama_endpoint: str apex_model_name: str kanana_model_name: str memory_threshold_gb: float # 기본 26.0 # 소스 sources: list[SourceConfig] # 12개 소스 설정 # 필터 tier1_quota: int tier2_quota: int tier3_quota: int tier3_hn_threshold: int # HN 업보트 임계값 tier3_reddit_threshold: int # Reddit 업보트 임계값 blacklist_keywords: list[str] # 보관 기간 processed_items_retention_days: int # 30 run_history_retention_days: int # 90 error_log_retention_days: int # 30 health_check_retention_days: int # 7 # 핫뉴스 hot_hn_threshold: int # 200 hot_reddit_threshold: int # 80
RawItem — Fetcher 출력 (미가공 뉴스) ▶
@dataclass class RawItem: url: str # 원문 URL title: str # 기사 제목 content: str | None # 기사 본문 (없을 수 있음) source_id: str # 소스 식별자 ("hackernews", "geeknews") fetched_at: datetime # 수집 시각 upvotes: int | None # Reddit/HN만 published_at: datetime | None # 발행일시 url_hash: str # SHA256(url)
NewsItem — 중복 제거 + 언어 감지 완료 아이템 ▶
@dataclass class NewsItem: url: str title: str content: str | None source_id: str fetched_at: datetime upvotes: int | None published_at: datetime | None url_hash: str lang: str # "ko" | "en" — LanguageDetector가 추가
SummaryResult — 요약/번역 결과 ▶
@dataclass class SummaryResult: item_url: str # 원문 URL summary_text: str # 요약 텍스트 (한국어) original_lang: str # 원본 언어 ("ko" | "en") summarizer_used: str # 사용된 Summarizer 구현체명 translator_used: str | None # 한국어 소스는 None error: str | None # 에러 발생 시 메시지
MemoryStatus — 메모리 상태 (Literal 타입) ▶
MemoryStatus = Literal["local_llm", "claude_fallback"]
SubscriberEvent — 구독 이벤트 ▶
@dataclass class SubscriberEvent: chat_id: str # 텔레그램 chat_id username: str | None # 사용자명 event_type: str # "subscribe" | "unsubscribe" occurred_at: datetime # 이벤트 발생 시각 update_id: int # Telegram update_id (중복 방지)
SendResult — 텔레그램 전송 결과 ▶
@dataclass class SendResult: total: int # 전체 전송 대상 수 success_count: int # 성공 건수 failed_chat_ids: list[str] # 실패한 chat_id 목록 errors: dict[str, str] # chat_id -> 에러 메시지
PipelineResult — 파이프라인 실행 결과 ▶
@dataclass class PipelineResult: run_at: datetime # 실행 시각 fetched_count: int # 수집 건수 dedup_count: int # 신규 아이템 수 filtered_count: int # 필터 통과 수 summarized_count: int # 요약 완료 수 sent_count: int # 전송 성공 수 elapsed_seconds: float # 총 소요 시간 memory_status: MemoryStatus # 메모리 상태 has_error: bool # 에러 발생 여부 error_summary: str | None # 에러 요약
CleanupResult — 데이터 정리 결과 ▶
@dataclass class CleanupResult: processed_items_deleted: int # 30일 초과 삭제 run_history_deleted: int # 90일 초과 삭제 error_log_deleted: int # 30일 초과 삭제 health_check_deleted: int # 7일 초과 삭제 cleaned_at: datetime # 정리 시각
